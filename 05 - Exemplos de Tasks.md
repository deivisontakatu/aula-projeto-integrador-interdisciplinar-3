# 🎫 Issue vs Draft

## 📝 Draft Issue

Usado para:
- Ideias iniciais
- Brainstorm
- Possíveis melhorias
- Itens ainda não refinados

Não possui:
- Número oficial
- Vinculação com PR
- Milestone
- Assignee formal

## 🎫 Issue Oficial

Usado para:
- Tasks formais
- Entregáveis da sprint
- Artefatos avaliativos
- Produção técnica real

Possui:
- Número (#)
- Labels
- Milestone
- Assignee
- Histórico
- Vinculação com commits e PR

📌 Regra do PI3:  
Toda task da Entrega 1 deve ser criada como Issue oficial.

# 🧾 Template Oficial de Issue

Utilizar este padrão para todas as Issues:

---

## 🎯 Objetivo
Descrever o propósito claro da tarefa.

## 🧠 Contexto
Explicar relação com a arquitetura geral e com a Entrega 1.

## 📦 Entregável
Descrever exatamente o que será produzido:
- Documento
- Diagrama
- Protótipo
- Especificação
- Modelo

## 🔗 Dependências
Listar outras Issues relacionadas.

## ✅ Critérios de Aceite
- [ ] Documento estruturado
- [ ] Diagrama anexado
- [ ] Revisado pelo time
- [ ] Validado pelo Scrum Master

## 👤 Responsável
Nome do membro da equipe.

## 🏷 Labels
iot / backend / cloud / ux / ivh / scrum / etc.

# 📊 Boas Práticas

- Uma Issue = Um entregável claro
- Evitar Issues genéricas
- Sempre anexar evidências (PDF, imagem, link Figma)
- Utilizar commits referenciando a Issue (#ID)
- Manter histórico para base do artigo científico
